The SADL Import Template Language (SITL)

Last revised 3/9/2022. Contact us.

Introduction

SADL and SadlServer support importing tabular data to create an OWL file (see CSV Tabular Data Importer). The SADL Import Template Language (SITL) is an Xtext grammar and an Eclipse-based editor that support creation and maintenance of tabular data import template files using a controlled-English grammar. The SITL grammar is consistent with the SADL grammar and hyperlinks semantic model concepts to their definitions in SADL or OWL files. Of particular interest, and the primary reason for creating SITL, the graph patterns that establish the mapping of tabular inputs to generated triples are expressed in exactly the same grammar and syntax used in SADL to define the semantic models.

By way of a summary, here are the assumptions underlying the tabular data importer.

  1. Each row of data in the tabular input can be processed as a single input to the template file and will result in a set of triples in the resulting OWL model.
  2. The elements in each row of data are referenced by the name of the column or by a sequential capital letter column designator, e.g., A, B, C, .., Z, AA, etc. This column name or column letter acts as a named variable which is bound to the corresponding table value, row by row.
  3. The table value, referenced by column name or column letter, may be transformed to create a new data value bound to a new named variable. Similarly, the value of a variable generated by a transformation may be further transformed to generate the value of another new variable.

Validation

Validations are expressed as SADL expressions. These validations may be type checks or may be equations which return a boolean value, in which case a return of true passes and a return of false fails. Here are some examples of validation statements.

  1. validate: ^A is localname(Rectangle) or ^A is localname(Circle). // The value in column A must match one of these two class local names
  2. validate: B is a decimal and B >= 0. // B must be a non-negative decimal number
  3. validate: C is a decimal and C >= 0. // C must be a non-negative decimal number
  4. validate: (B + C) < 10000. // The sum of B and C must be less than 1000
  5. validate: F is a boolean or F is a nonNegativeInteger. // F is either boolean or an integer >= 0

Transformations

Transformations are accomplished by SADL expressions. These expressions may use grammar-supported operations, such as math operations, or may use equations elsewhere defined and imported. Here are some examples of transformation statements.

  1. transform: C2 = C + 1.
  2. transform: C3 = strConcat("T", C2). // prepend "T" to the value of C2
  3. transform: C3 = "T" + C2. // another way to do the same thing
  4. transform: D2 = split(D, ":"). // split the value of D into a list on delimiter ":"
  5. validate: D2 is a string List. // check that the result is a list of strings

Mapping Tabular Inputs to Triples

The mapping of the data in a tabular input to triples in the generated semantic model uses the same syntax as concept definitions in a SADL model. In other words, SADL statements are used in SITL but references can be to concepts (individuals, classes, and properties) in imported semantic models or references can be variables corresponding to a column in the tabular data or to a derived value resulting from a transformation.

To illustrate we will use the same example found in the first part of the CSV Importer documentation. This is the domain model. (Note that this model is slightly updated to reflect the SadlImplicitModel's UnittedQuantity class.)

uri "http://sadl.org/TGModel.sadl" alias tgmodel.

Observation is a type of UnittedQuantity,
    described by timestamp with values of type dateTime,
    described by observer with values of type string.

Speed is a type of Observation.
Power is a type of Observation.

Turbine is a class,
    described by drives with a single value of type Generator,
    described by speed with values of type Speed.

Generator is a class,
    described by power with values of type Power.

This is the SITL template generator file to generate a template for the data table in the CSV Importer example, with columns "TurbineID", "Speed", "Power", and "Time".

uri "http://sadl.org/TGTemplate.sitl" alias tgt.

import "http://sadl.org/TGModel.sadl".

validate: ^A is a string.
validate: B is a decimal.
validate: C is a decimal.
validate: D is a dateTime.

Transform: TID = "T" + ^A.
Transform: GID = "Gen" + ^A.

TID is a Turbine with speed
    (a Speed with ^value B, with unit "RPM", with timestamp D).
TID drives
    (a Generator GID with power
        (a Power with ^value C, with unit "MW", with timestamp D)
).


Groups

Groups allow more control over processing. For example, suppose that we have a table of data in which column A has the type of a shape, Rectangle or Circle,  column B has a first dimension and column C a second dimension, if there is one. Now suppose that we create two derived variables, A1 and A2, each of which is the value of the element in column A. We can then do validations with "skip group" to cause the appropriate triples to be inserted into the generated OWL.

transform: A1 = ^A.

transform: A2 = ^A.

validate: A1 is localname(Rectangle) skip group.

validate: A2 is localname(Circle) skip group.

{

    an A1 with height B, with width C.

    the A1 has additionalInfo "this is a Rectangle".

}

{

    an A2 with radius B.

    the A2 with additionalInfo "this is a Circle".

}

If the validation of A1 fails (it is not "Rectangle") then the first group will be skipped. If the validation of A2 fails (it is not "Circle") then the second group will be skipped.

Control Statements

Control statements determine how the tabular data importation proceeds. They include the following, shown in SITL syntax.

  1. Incremental. // add the generated triples to the existing model if such exists
  2. Infer: chunk 20 parallel 9. // do inference over 20 rows of data at a time, with up to 9 parallel threads
  3. Log: "ImportLog.log". // generate a log file containing an ordered record of each triple generated, along with validation information
