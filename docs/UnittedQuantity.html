Unitted Quantities in SADL

Last revised 5/16/2022. Contact us.

Introduction

Quantitative measurement is an important concept in many modeling problems. A quantitative measure has a numerical value and usually has a unit associated with it, which gives meaning to the number by relating the quantitative value to some standard amount, e.g., 16 feet. In many cases, giving only a number is either completely ambiguous or requires that an assumption be made about the unit of measure. Furthermore, derived values often have units associated with them as well. Whether a direct measurement or a derivation therefrom, a value along with its associated unit is what we will refer to as a unitted quantity.

To illustrate, suppose that we have a domain model defining the data type properties age and weight.

Person is a class described by age with values of type decimal,
        described by weight with values of type decimal.

We could use these properties in instance data statements like

George is a Person with age 23, with weight 165.

Jim is a Person with age 2, with weight 9.5.

An observer with experience would be able to guess, at least in the context of English units, that George's age is 23 years  and his weight is 165 lbs and that Jim's age is 2 days or 2 months, but not 2 years, and his weight is 9.5 lbs. However, for quantitative modeling we would often like to avoid the ambiguity by making the units explicit.

Units in SADL

The SADL grammar allows the statements above to be expressed more specifically as

George is a Person with age 23 years, with weight 165 lbs.

Jim is a Person with age 2 days, with weight 9.5 lbs.

In other words, the SADL grammar supports placing a unit designation after a numeric value in such statements. If the unit has spaces, e.g., “feet per second”, it must be placed in quotes.

In order to translate statements such as these into OWL, we must have a meta-model of quantities with values and units. The model SadlImplicitModel.sadl provides such a meta-model. Note that the caret in front of "value"  in the definition below is necessary to escape the value keyword in the SADL grammar and indicate that in this case it is a user-defined concept.

UnittedQuantity is a class,
       described by ^value with values of type decimal,
       described by unit with values of type string.

Domain models can sub-class UnittedQuantity to create more specific classes of quantitative measures and derivations. This allows modelers to create more complex models of UnittedQuantities, see QUDT as an example. If transitive closure is enabled by the reasoner, instances of all sub-classes of UnittedQuantity should behave in the expected way as they will also be instances of UnittedQuantity.

The translation of the instance statements above from SADL to OWL is made as if the SADL statements were

                 George is a Person
                      with age (a UnittedQuantity with ^value 23, with unit "years"),
                      with weight (a UnittedQuantity with ^value 165, with unit "lbs").

Jim is a Person

                     with age (a UnittedQuantity with ^value 2, with unit "days"),
                     with weight (a UnittedQuantity with ^value 9.5, with unit "lbs").

Experience has shown that some applications prefer that the SADL statements include units, but that the translation to OWL does not include units. The translation of SADL UnittedQuantity expressions to OWL can be controlled to achieve either result by a SADL preference setting. If the preference “Ignore Unitted Quantities (treat as numeric only) during translation” is checked, the translation will happen as if the units were not present in the SADL statements. If this preference is not checked, the definition of UnittedQuantity from the SadlImplicitModel.sadl is used and the expression, e.g., 23 years, is translated to a blank node instance of the class UnittedQuanity with properties value and unit, as illustrated by the previous statements.

When a property such as age or weight in the examples above is defined and given a range, the range in the OWL model should be consistent with the planned usage. For example, consider this definition of Person.

         Person is a class described by age with values of type UnittedQuantity,
             described by weight with values of type UnittedQuantity.

When the preference to ignore unitted quantities is not checked, each property in the statement above  (age, weight) is translated to an owl:ObjectProperty with the range UnittedQuanitity. If ignore unitted quantities is checked, each property is translated to an owl:DatatypeProperty and the range is the range of the implicit model value property, which is XSD decimal.

Unitted Quantities in Expressions

Unitted quantities and properties whose ranges are unitted quantities may also be included in SADL expressions used in rules, queries, and tests. In particular, unitted quantities can appear in math operations such as addition, subtraction, multiplication, division, and exponentiation, and in comparison operations such as less than, greater than, etc.

Using a rule to illustrate, and building on the examples above and with UnittedQuantity specified as the range of age, consider the following rule.

    Rule AdultRule: if p is a Person and p has age  >= 18 years then p is an Adult.

Without assuming any automatic conversion from one unit of time to another (i.e., that the values of age in the knowledge base are expressed in years), and assuming the presence of a numerical comparison operator in the reasoner, the rule below is equivalent to the rule above.

Rule AdultRule: if p is a Person and p has age uq and unit of uq is "years" and ^value of uq >= 18  then p is an Adult.

In this rule the variable uq is of type UnittedQuantity, obtained from the range of age. The unit  is years condition ensures that values of age that are in other units of time do not inappropriately fire the rule. The value property is extracted from the UnittedQuantity bound to uq so that it can be used in the numerical comparison.

Note that for expressions in rules and queries, a limitation of the SADL grammar is that the unit designation given immediately following the value expression cannot be a variable; it must be a string or quoted string. Of course a variable may be of type UnittedQuantity, and if the unit property is explicit in the expression then its value may be a variable. In the second rule above, the value 'years' in expression unit of ug is 'years' could be a variable with the appropriate string value for unit bound to it. However, in the first rule, in the expression 18 years, a variable could not be used after the value 18 in place of years.

In order for the first rule above to work with a reasoner supporting numerical built-in functions as operators, the range of age and the value and unit of the right side of the greater than or equal comparison operator must be preserved in the Intermediate Form, passed on to the reasoner-specific translator for correct translation, and be represented in the final rule given to the reasoner for inference (see SADL Translator/Reasoner Plug-in Pairs). There are two different possible approaches to accomplishing the translation. One approach is to expand the details of each element of type UnittedQuantity to make the value and unit triples explicit, as they were in the second rule above. The other approach is to enhance the built-in math operators that work on unitted quantities, such as ">=", to handle arguments of type UnittedQuantity. For the Jena-based reasoner/translator pair, SADL uses both approaches are possible. Which is used depends upon the preference setting "Expand UnittedQuantity in translation". If this preference is checked, the value and unit properties of UnittedQuantity will be expanded and made explicit in translation. If it is not checked, where possible UnittedQuantity instances will be passed directly into math and comparison operators.

Computational Operators Supporting Unitted Quantities (when using Jena Reasoner Plug-in and SADL-provided Built-in Functions)

Computational operators with UnittedQuantity inputs generate a new UnittedQuantity output. The unit of the output depends upon the type of operator. Computational operators include the following.

  1. Product (*) -- if all of the arguments are instances of UnittedQuantity, the result of the multiplication will be a new instance of UnittedQuantity whose value is the product of the values of the multipliers and whose unit is the product of the units of the multipliers. For example, 2.0 ft x 3 .0ft will yield 6.0 "ft*ft". The first argument can be a UnittedQuantity and the second a number only, e.g., 40 mph * 2 will yield 80 mph.
  2. Sum (+) -- if all of the arguments are instances of UnittedQuantity and have the same units, the result will be a new instance of UnittedQuantity whose value is the sum of the values of the multipliers and whose unit is the unit of the addends. For example, 2.0 ft + 3 .0ft will yield 5.0 ft.
  3. Quotient (/) -- if both arguments are instances of UnittedQuantity, the result of the division will be a new instance of UnittedQuantity whose value is the quotient of the division and whose unit is the quotient of the dividend and divisor. For example, 12.0 ft / 3 .0 sec will yield 4.0 "ft/sec". The first argument can be a UnittedQuantity and the second a number only, e.g., 80 mph / 4 will yield 20 mph.
  4. Difference (-) -- if both arguments are instances of UnittedQuantity, x - y  translates to condition unit of x = unit of y and numerical result value of x - value of y.
  5. Exponentiation (^) -- only the first argument can be an instance of UnittedQuantity and the result will be the value of the first argument raised to the power of the second argument. For example, 4 ft ^ 3 will yield 64 ft*ft*ft.

Note that the product (*) and sum (+) operators can take any number of input arguments of type UnittedQuantity and a single input argument which is a list of typeUnittedQuantity. The remaining computation operators listed above are binary. The sum and difference operators require that the unit of both arguments are the same.

Comparison Operators Supporting Unitted Quantities

All of the comparison operators listed below are binary operators and the unit must be the same for both arguments. There is no computation of a UnittedQuantity so complete expansion of triple patterns can be done at translation. The condition is added to the expressions that the unit of both arguments must be the same. The value of each argument is passed to the numical comparison operator as numeric input.

  1. GE (>=) -- if one argument is an instance of UnittedQuantity then both arguments must be instances of UnittedQuantity; x >= y  translates to condition unit of x = unit of y and boolean result value of x >= value of y.
  2. GreaterThan (>) -- if one argument is an instance of UnittedQuantity then both arguments must be instances of UnittedQuantity; x > y  translates to condition unit of x = unit of y and boolean result value of x > value of y.
  3. LE (<=) -- if one argument is an instance of UnittedQuantity then both arguments must be instances of UnittedQuantity; x <= y  translates to condition unit of x = unit of y and boolean result value of x <= value of y.
  4. LessThan (<) -- if one argument is an instance of UnittedQuantity then both arguments must be instances of UnittedQuantity; x < y  translates to condition unit of x = unit of y and boolean result value of x < value of y.

Other Operators Supporting Unitted Quantities

There are some additional operators that can take UnittedQuantity arguments. Although these operators do not create new UnittedQuanity instances, they do return a UnittedQuantity instance and are handed by the built-in operators themselves.

  1. Max (max) -- if one argument is an instance of UnittedQuantity then both arguments must be instances of UnittedQuantity; max(x, y  translates to condition unit of x = unit of y and numerical result max(value of x, value of y).
  2. Min (min)-- f one argument is an instance of UnittedQuantity then both arguments must be instances of UnittedQuantity; min(x, y  translates to condition unit of x = unit of y and numerical result min(value of x, value of y).

Unitted Quantities in Rules

The math and comparison operators used by the JenaReasonerPlugin have been extended to support UnittedQuantity arguments. Therefore, the preference setting "Expand UnittedQuantity in translation" can be used to determine the approach taken to translating and using these operators. Other reasoner/translator pairs can implement direct processing of UnittedQuantity arguments if desired.

Unitted Quantities in Queries

SPARQL queries are mostly opaque strings in SADL and specify, in the where clause, the graph patterns to be matched. They are passed directly to the selected reasoner for evaluation. However, comparison operators can appear in the filters of query statements. When simple queries are expressed in the SADL query and expression language, the conditions can include comparison operators from the list above. These will be expanded in the resulting SPARQL query string, yielding additional triple patterns to be matched and an additional filter condition on units. In other words, all expressions involving UnittedQuantity will be expanded for SADL queries.

Unitted Quantities in Query Results

When using the Jena Reasoner Plug-in, a query result that contains instances of UnittedQuantity will be automatically expanded to show the value and unit of each UnittedQuantity result. If the UnittedQuantity instance is a named instance, that name will also be included. If it is a bnode, only the value and unit but not the system-generated bnode identifier will be shown in the query results for easier human understanding.

Unitted Quantities in Tests

Comparison operators can be used in SADL test statements.
