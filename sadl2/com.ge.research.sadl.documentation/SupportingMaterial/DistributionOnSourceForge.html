Distribution of SADL on SourceForge

Building the Release

  1. Update the version number in the Overview tab of the plugin.xml file of the com.ge.research.sadl and com.ge.research.sadl.ui projects. Save changes.
  2. Update the version number in the Overview tab of the feature.xml file of the com.ge.research.sadl.feature project. ("qualifier" will be after the version number). Save.
  3. On the same tab as #2, click on "Synchronize" and select the first option, "Synchronize versions on build...", then click on Finish. Save changes to feature.xml.
  4. Open "site.xml" in the com.ge.reaserch.sadl.update project. Update to the latest in head from CVS to restore "quantifier" as the end of the version.
  5. If a new release, use "Add Feature..." to add the new feature. (Remove the old feature?). Click on "Build All". (Note: check changes to site.xml into CVS BEFORE doing Build All as that replaces "quantifier" with the actual build.)

Notes

  1. The "bin" folders must be checked in the build.properties files to generate the content in the distribution jars. Likewise for "lib" if jars are to be included.
  2. It may be necessary to delete the content of the update project, including the features and plugins folders, to get a complete "Build All" result.

SADL Update Site

Using Putty PSFTP

  1. Using an appropriate FTP client such as Putty PSFTP, change your local directory to the local update project folder. (You can right-click on the Eclipse project and select properties to see the actual disk location of the Eclipse project.)
  2. Open a connection to
              <username>,sadl@web.sourceforge.net
    where <username> is your SourceForge userid.
  3. Change the SourceForge directory to the update foler: cd htdocs/update
  4. Transfer each file in the update folder (site.xml, content.jar, atrifacts.jar) to the server. (This must be the site.xml file as modified by the Build All.)
  5. Change both the local and remote current folder to the features folder:
    lcd features
    cd features
  6. Transfer the com.ge.research.sadl.feature_xxxxxxxx.jar file to the server (xxxxx will change for each build of the update site)
  7. Change both the local and remote current folder to the plugins folder
  8. Transfer the two jar files in the plugins folder to the server

Using WinSCP

  1. Start WinSCP and enter <sf_username>,sadl@web.sourceforge.net as the host, enter username and password, and click on Login.
  2. Navigate local file system on the left, SourceForge Web on the right, copying files as desired. (Copy same files as in #3-8 above.)

SADL Web Site

To transfer new content to the SourceForge Web site for SADL do the following.

  1. Similar to the first two steps in the section above, use an appropriate FTP client and connect with the local directory the WebSite folder containing the Web content--the folder containing "index.html".
  2. The remote folder should be the default, "htdocs".
  3. Transfer all files in the WebSite folder to SourceForge. For example, with PSFTP, use the "mput *.*" command.
  4. Change to each local folder and to the corresponding remote folder one by one, recursively (walking down to the leaf folders) and repeat until all content has been uploaded.
  5. Note: only changed files need to be uploaded.
