Typed Lists in SADL

Introduction

RDF offers a List construct as a type of ordered collection. However, the rdf:List-related concepts are used in the serialization of OWL itself, and so are not usable to capture domain knowledge as ambiguities would arise regarding what was part of the OWL encoding and what was part of the data. In addition, rdf:List constructs do not distinguish between literal values and instance values, let alone provide typing useful for inference and type checking.

Two approaches seem to offer themselves. One is an implementation of List constructs in OWL, of which two are available in the public domain: the collections ontology (Reference 3) and an earlier OWL sequence patterns (References 1 and 2). The second approach is a truly typed List construct.

The Collections Ontology

 

Type Lists from Scratch

A list is an ordered set of values. When order doesn't matter, a multi-valued property can be used, e.g., if Genny is a Person who has two pets.

    Genny is a Person with pet Lassie, with pet Spot.

However, sometimes order matters. In OWL, to which SADL models are translated, lists are sometimes handled using rdf:List. However, the rdf:List construct places an OWL model in the OWL Full realm and there is no typing of list elements. For these reasons, SADL creates its own OWL representation of typed lists. The SADL language provides grammar to specify that the range of a property is a List, or multiple Lists, of values of a particular type. The elements of the list can be literals, as would be the values for a datatype property, or they can be individuals, as would be the values of an object property. Here are examples class and property definitions using each types.

    StandardValueList is a class, described by elements with a List of values of type float.

     Person is a class, described by pet with values of type Pet, described by prioritizedPets with a List of values of type Pet.

The second definition allows an instance declaration that illustrates the difference between a multi-valued property and a property range which is a list.

    Genny is a Person with pet Lassie, with pet Spot, with prioritizedPets [Spot, Lassie].

If we were to ask "Genny has pet ?x" we would get back "Lassie" and "Spot", in indeterminate order. But if we ask "Genny has prioritizedPets ?x" we will get back the list ["Spot", "Lassie"]. In the first case the match is two triples: "Genny has pet Lassie" and "Genny has pet Spot" and there is no explicit order. In the latter case we would get back one triple: "Genny has prioritizedPets [Spot, Lassie]".

The implementation of typed lists in SADL depends upon the creation, in OWL, of structures implied by the grammar in SADL but not explicitly present. Each time the range of a property is a "List of values of type ...", a class unique to the property must be created. Assuming that this typed list is the only class in the range of the property, the implementation is fairly straight forward. For the two examples above, respectively:

  1. There exists a class "List-Member-elements", which is the range of "elements", and which is the domain of two properties:
    • "elements-float", with range float
    • "elements-float-next", with range "List-Member-elements"
  2. There exists a class "List-Member-prioritizedPets", which is the range of of "prioritizedPets", and which is the domain of two properties:
    • "prioritizedPets-Pet", with range Pet
    • "prioritizedPets-next", with range "List-Member-prioritizedPets"

If the range of the property having the "List of values" is a union with other values, what do we do?

Retrieving Lists via SPARQL and SADL Query

Retrieving Lists with SPARQL will require knowledge of the underlying implementation of Typed Lists in SADL and can be done using recurring property chains.

Retrieving Lists with SADL query can hide the details because the translator can look at the range of the property and, seeing that it is a List of values, can generate the SPARQL equivalent using the recurring property chain.
