SADL-IDE Development Environment Setup and Build Instructions

This document describes how to set up the SADL-IDE development projects in Eclipse and then build a working set of plug-ins that enables changes to the grammar and/or the implementation code. There are two levels of development possible. If there is no need to modify code in the two projects com.ge.research.sadl.reasoner and com.ge.research.sadl.jena projects (if the primary interest is in the Xtext UI and builder code), then Maven is not required and these two projects do not need to be checked out. The jar files corresponding to these two projects are checked in under the projects requiring them, including source code for debugging. However, if these projects are part of the scope of the development environment Maven must be installed and used to build  the projects.

Prerequisite

  1. Eclipse Kepler properly installed and configured with Xtext.
  2. If you wish to be able to run Maven from the command line (as opposed to only within Eclipse, install the latest Maven (3.0.3) from http://maven.apache.org/download.html.

Build Steps

  1. Install Eclpse Kepler with Xtext 2.4.3 via the Eclipse Update Manager from the update site: http://download.eclipse.org/modeling/tmf/updates/releases 
  2. Install the Maven plugin, m2eclipse, via the Eclipse Update Manager from the update site: http://m2eclipse.sonatype.org/sites/m2e
  3. If you use other than default repositories for Maven, configure m2eclipse (Window -> Preferences : Maven -> User Settings ). Set the User Setting file to point to the settings file in the Maven installation (Prerequisite #2). It is likely in C:\Program Files\Apache Software Foundation\apache-maven-3.0.3\conf\settings.xml (Windows) or /home/<user>/.m2/settings.xml (Linux).
    Now edit this file to set up proxies and mirrors as follows, similar to the following:
    <settings>
      <proxies>
       <proxy>
          <active>true</active>
          <protocol>http</protocol>
          <host>proxy.research.ge.com</host>
          <port>8080</port>
          <nonProxyHosts>localhost|*.ge.com|3.72.112.153</nonProxyHosts>
        </proxy>
      </proxies>  
      <mirrors>
        <!-- Go to our Nexus for everything from central, c5, flex, jboss, and v99 -->
        <!-- For SSI Development, use the GE Energy SSE Nexus
    Repository -->
        <mirror>
          <id>ssi-nexus</id>     
          <mirrorOf>central,java.net,log4j,maven2-repository.dev.java.net,open.iona,open.iona.m2</mirrorOf>
          <name>GE Energy's Nexus Repository</name>
          <url>http://3.72.112.153/nexus/content/groups/public</url>
        </mirror>
      </mirrors> 
    </settings>
  4. Install the Service Loader Maven Plug-in (not part of Maven repository so requires custom install)
    • Download the service loader plug-in from: https://github.com/francisdb/serviceloader-maven-plugin
    • Add to your local Maven repository
      • If using Maven outside of Eclipse, unzip to a directory of your choice and run "mvn install"
      • If using Maven within Eclipse, import the project as a Maven project within Eclipse and then do a Run As...-> Maven Install on the project
  5. Checkout the following projects from SADL (sadl.cvs.sourceforge.net).
    • com.ge.research.sadl.documentation
    • com.ge.research.sadl.feature
    • com.ge.research.sadl.updatesite
    • com.ge.research.sadl
    • com.ge.research.sadl.generator
    • com.ge.research.sadl.jena                (not needed for limited development environment)
    • com.ge.research.sadl.reasoner        (not needed for limited development environment)
    • com.ge.research.sadl.tests
    • com.ge.research.sadl.ui
    Any CVS client, e.g.,  SmartCVS,  to check out the projects. If you use Eclipse to check out the projects you will need to then delete the Maven projects (...reasoner, ...jena) but do not delete the content.
  6. If you did not check out the projects within Eclipse,  Import the remaining projects (except reasoner and jena) as "Existing Projects"
  7. Build those projects (or check build automatically).
  8. Import the reasoner and jena projects into Eclipse as Maven projects using (File->Import->Maven Existing Maven Projects) pointing to their respective folders from the checkout.
  9. If the checked out .classpath files do not resolve all references in the reasoner and jena project, update the Java build path (Project->Properties->Java->Build Path) to add the JRE System Library and Maven dependencies.  Also set the Maven active profile to pom.xml.
  10. Right click on the com.ge.research.sadl.reasoner project and Select Run as -> Maven Install. Then refresh the "target" folder and copy the two jar files to the lib folder of the ...sadl and ...sadl.ui projects. This must be done any time something changes in the reasoner project.
  11. Right click on the com.ge.research.sadl.jena  project and Select Run as -> Maven Install. Refresh and copy from the "target" folder to the lib folder of the ...sadl and ...sadl.ui projects. Repeat any time a change is made in the jena project.
  12. In the com.ge.research.sadl project navigate to src/com.ge.research.sadl right click on the GenerateSadl.mwe2 file and click on Run as -> MWE2 Workflow. This will build the necessary files in ...sadl and ...sadl.ui projects from the SADL grammar defined in src/com.ge.research.sadl.Sadl.xtext.
  13. Right click on com.ge.research.sadl.ui project and click Run as -> Eclipse Application

Creating Reasoner/Translator Plug-in Projects that as Maven Projects

It is desirable to have a reasoner/translator plug-in project be a Maven project without any Eclipse dependencies so that it can be packaged with the SadlServer or other service providers. It is desirable to have a reasoner/translator plug-in project be an Eclipse plug-in project so that it will automatically be included and the implementation classes discovered by the Java Service loader. While one can meet these competing demands by having two projects, the disadvantage is that the code is duplicated. The following approach allows a single project to exist.

  1. Create a Maven project with source in src/java/main, etc., and a pom.xml file. The output should be target/classes.
  2. Place the plug-in Activator class in the project but exclude it in the pom.
  3. Create a META-INF folder with a MANIFEST.MF file with all of the appropriate plug-in information.
  4. Run Maven -> Update Project to generate a classpath from the pom file.
  5. Run Maven install to build source and binary jars in the target folder.
  6. Open the binary jar and extract the META-INF/services folder and subfolders to the project's META-INF folder.
  7. Open the project properties and remove, from the sources tab, the exclusion of the Activator class.
  8. Clean the Eclipse project if it doesn't automatically rebuild.
